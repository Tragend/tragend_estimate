# Tragend Loop — 開発体制マニュアル（正本）

> 株式会社Tragend のAI自律開発体制。本プロジェクト（tragend_estimate）で動かして検証し、**他プロジェクトへ移植できる会社の再利用資産**として確立する。
> 思想的土台: [Loop-Engineering_AI開発体制の基盤.md](../../../../01_company/Loop-Engineering_AI開発体制の基盤.md)（Addy Osmani）。確定: 2026-06-22。

---

## 1. 全体像

```
[人間ゲート①] RD/SD確定（料金含む）        ← ループ起動の前提
      │
      ▼
[Phase 0] 検証ゲートの構築（←自律ループより先に作る）
      Vitest／体制マニュアル／REVIEW_RULES.md／BACKLOG.md／Workflow雛形／memory正本化
      （後追い：CI・GitHub private repo）
      │
      ▼
[フェーズA〜D] 自律ループ（1フェーズ＝1 Workflow・手動キック）
      Planner → 実装 → クロスレビュー3レンズ → 機械ゲート(test/lint/build) → 差し戻し → 全合格でPR作成 → ★停止
      │
      ▼
[人間ゲート②] PR承認 → デプロイ
```

人間の関与は **①設計（料金確定）** と **②PR承認** の2点だけ。途中（実装〜レビュー〜テスト緑〜PR）はWorkflowが自動で回す。

## 2. 確定した方針

| 項目 | 確定内容 |
|---|---|
| クロスレビュー | **Claude多層**（実装と採点を別ティア・別ペルソナで分離。書いた本人は採点しない）。他社モデル・CodeRabbitは将来 |
| 自律レベル | **開発フェーズ手動キック**。ユーザー起動→自動完走→**PRで必ず停止** |
| マージ判断 | PR停止→ユーザー承認→デプロイ |
| レビュー役 | **既製品のまま**（`code-reviewer`／`security-auditor`／`test-engineer`）＋ナレッジmd差し込み |
| エンジン | `Workflow`ツール（決定的fan-out＋採点パイプライン）。全フェーズで再利用 |
| worktree並列 | **今やらない**（直列・ブランチ1本）。将来の「いつかやる」枠 |

## 3. ナレッジの差し込み（レビュー役の味付け）

レビュー役は既製品のまま、Workflowが各役に「次を読んでから採点せよ」と**プロジェクト固有のmdを注入**する。ナレッジ更新＝mdの編集だけで済む（移植性の核）。

| ナレッジ | 置き場所 | 読まれ方 |
|---|---|---|
| プロジェクト全体ルール | `CLAUDE.md`（repo直下） | 毎回自動でコンテキストに入る |
| レビュー観点の独自ルール | `docs/loop/REVIEW_RULES.md` | Workflowが各レビュー役に指示 |
| 設計の正本 | `docs/SD/SD.md`・`docs/RD/RD.md` | 同上 |

## 4. State / Memory（文脈リセット対策）

| 層 | 用途 | 実体 |
|---|---|---|
| 進捗の栞 | 残作業・受入基準・status | `docs/loop/BACKLOG.md`（本PJ正本） |
| 決定の記憶 | 体制・料金方式など恒久的事実 | Claude Code の `memory/`（セッション横断で自動想起） |
| 全社俯瞰 | 会社レベルの優先タスク | `/Users/know/Tragend/TASKS.md` |

## 5. トリガー

ユーザーがコマンド／自然文でキックする＝トリガー。当面は「**フェーズAをやって**」のような自然文で起動→私がWorkflowを立ち上げる。定型化したら専用スラッシュコマンド（例 `/run-phase A`）を自作する。

## 6. 残しておく人間の責任（記事の核・自動化しても消えない）

1. **検証責任** — 無人ループは無人ミスを生む。機械ゲート（test赤緑）は必須。
2. **理解の負債** — AI量産コードを読むことは続ける。
3. **思考停止リスク** — 自動化に判断を明け渡さない。
