# REVIEW_RULES — tragend_estimate 固有のレビュー観点（ナレッジ差し込み用）

> Tragend Loop のクロスレビュー時、各レビュー役（code-reviewer / security-auditor / test-engineer）に
> 「`docs/SD/SD.md` と本ファイルを読んでから採点せよ」と注入する。既製品レビュー役をこのプロジェクト仕様に味付けする正本。
> 設計の根拠は [docs/SD/SD.md](../SD/SD.md) / [docs/RD/RD.md](../RD/RD.md)。矛盾時はSD/RDが優先。

---

## 鉄則（違反は即不合格）

1. **AIは金額（円）を返してはならない。金額計算は100%コード側（`src/lib/estimate/pricing.ts`）で決定的に行う。**
   AIの出力スキーマ `EstimateRequirement` に円・金額フィールドが混入していないか確認（ハルシネーション防止＝最重要の安全装置）。SD §3.1。
2. **内部ラベルを顧客向け成果物に出さない。** `size`(S/M/L/XL)・`isHighRisk`・「地雷」・「×1.5」は社内専用。顧客が見るのは機能名・人日・金額・合計のみ。地雷割増は人日に溶け込ませる。SD §1.4。
3. **構想書は「叩き台（ドラフト）」。** 構想書PDFに「確定要件は商談で人間が定める」旨の必須注記があるか。AI出力にそのまま則って開発する設計になっていないか。RD §3.1。
4. **秘密情報・LLM呼び出し・DB操作は全てサーバ側**（Server Actions）。APIキーがクライアントに漏れる経路が無いか。`.env` がGit追跡対象になっていないか。
5. **Supabaseは公開APIから書き込みのみ・読み取り不可（RLS有効）。** 公開経路から `quotes` を読める設定になっていないか。SD §3.4 / §4。

## 金額ロジック（pricing.ts）レビュー時

- 人日単価・固定費・size→人日テーブル・税率・地雷係数は **SDの確定値と一致**しているか（SD §1.1 / §3.2）。
  ※ ただし**料金は人間ゲート①で最終確定**するため、未確定の数値を勝手に変えない。SDが正本。
- 不正な `size` で NaN（金額崩壊）にならず明示的にエラーになるか。
- 下限 ¥500,000（固定費）が常に確保されるか。

## 一般観点（既製品レビュー役の標準軸＋本PJの強調）

- **correctness**: 金額計算・人日集計・税計算の境界（0機能・大量機能・全地雷）。
- **security**: 入力バリデーション・スパム/レート制限・同意取得・データ最小保持。納品物としてのセキュリティは「機能として計上」されているか（タダにしない）。SD §4(b)。
- **test/coverage**: 決定的ロジック（pricing）に単体テストがあるか。AI・PDF・メール等の外部依存はモック境界が切れているか。

## 機械ゲート（レビューと別に必須）

`npm test` 緑 ＋ `npm run lint` ＋ `npm run build` 通過。レビュー合格と機械ゲート通過の**両方**でPR作成。
