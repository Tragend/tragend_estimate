## 1. 価格ロジック ← いま設計中

### 1.1 基本式（確定：2026-06-23 ゲート①でレビュー確定）
```
開発費 = Σ(機能ごとの人日) × 人日単価(¥35,000)
固定費 = PM ¥150,000（常時）
       + PoC ¥100,000 / UI・UX ¥150,000 / テスト・QA ¥100,000 ← AIが案件性質で要否判定
小計   = 開発費 + 固定費
税込   = 小計 × 1.1
```
- 校正基準は **従来工数基準（バリューベース）**：人日は「普通のエンジニアの工数」で置き、AI活用の速さは値引きせず **粗利** として握る。
- 固定費は **定額だが項目ごとにAIが要否判定**（HP制作にPoC不要 等）。PMのみ常時計上＝実質の最低ライン。**人工的なフロアは設けない**。
- 実装は `src/lib/estimate/pricing.ts`（決定的）＋ `pricing.test.ts`（確定値を固定）。AIは固定費の要否(真偽)のみ返し、金額はコードが確定。

### 1.2 機能ごとの人日（確定テーブル）
- AIが各機能に **規模(S/M/L/XL)** を割り当て → 下表で人日へ変換（金額計算は決定的）。
- 確定テーブル（従来工数基準）: **S=1.0 / M=2.0 / L=3.5 / XL=6.0**。目安: S=ログイン級 / M=会員登録級 / L=管理画面級 / XL=決済連携級。
  ※ 旧 miryo帯（S=1.0/M=1.5/L=2.0/XL=2.5）はスプレッドが圧縮され概算が低く出るため、複雑・地雷機能の重みを増やして改定。

### 1.3 決めること（全確定）
- [x] **人日単価 → ¥35,000（確定）**
- [x] **固定費 → 定額・項目AI判定**：PM ¥150,000（常時）／PoC ¥100,000・UI/UX ¥150,000・QA ¥100,000（AI要否判定。QA既定ON）。フロアなし
- [x] **表示 → 単一値（税込）＋概算注記**（→ 1.4）
- [x] **運用費 → 自動見積は開発費のみ。別途注記＋詳細は商談**（→ 1.4）
- [x] **機能→人日の出し方 → AIが S/M/L/XL を判定 → 人日へ変換**（S=1.0／M=2.0／L=3.5／XL=6.0）
- [x] **有効期限 → 発行日から30日**（`QUOTE_DEFAULTS.validityDays`）

### 1.4 表示・注記（確定）
- 金額は **単一値（税込）を大きく表示**（レンジにしない）。工数単位は **人日（man-day）**。
- **概算ディスクレーマー**を見積書に必須記載。Tragend版ドラフト:
  > 本お見積もりはあくまで概算です。詳細なヒアリングのうえ、正式なお見積書を改めてご提出いたします。なお、本見積もりには開発費のみを含み、運用・保守費は別途お見積もりとなります。
- 名義: 株式会社Tragend
- **内部ラベルは顧客に出さない**：`size`(S/M/L/XL)・`isHighRisk`・「地雷」・「×1.5」は社内専用。顧客が見るのは **機能名・人日・金額・合計** のみ（地雷割増は人日に溶け込ませて見せる）。内部確認は CLI / Supabaseコンソール / 管理画面で。

### 1.5 見積もり安全装置（軽量・確定は商談で）
- 本システムは "実績づくり" が主目的。**確定金額は商談で Tragend（人間）が出す** ＝これが実質の防御。安全装置は作り込まない。
- 唯一の組み込み：**地雷カテゴリの人日割増 ×1.5**（決済・外部連携・AI/RAG・規制業種〔金融/医療〕等）。
- 最低ラインは **PM ¥150,000 常時計上＋開発費** で自然に確保（人工的なフロアは設けない＝HP制作等を安く出せる柔軟性を優先）。

## 3. AI抽出・データ・生成パイプライン

### 3.1 AIの出力スキーマ（keystone）
**原則：AIは「構造と規模」だけ返す。金額（円）は一切返さない → 金額計算は100%コード**（ハルシネーション防止＝安全装置）。

AIが返す構造化案件要件（Claude の tool use で型を固定）:
```ts
interface EstimateRequirement {
  projectTitle: string;    // 件名（AI生成）
  summary: string;         // 背景・ゴール要約（deck案件概要用）
  features: Feature[];     // AI導出の機能リスト
  fixedCosts: {            // 固定費の要否判定（PMは常時計上なので含めない・金額は返さない）
    poc: boolean;          //   検証(PoC)が必要か（HP制作等は false）
    uiUx: boolean;         //   UI/UX設計が必要か（画面ありは true）
    qa: boolean;           //   テスト・QAを計上するか（通常 true）
  };
  assumptions?: string[];  // 前提・除外（任意）
  needsHearing?: boolean;  // 大型/複雑 →「要詳細ヒアリング」推奨
}
interface Feature {
  category: string;        // 大分類（会員管理/決済/外部連携/AI/セキュリティ等）
  name: string;            // 機能名
  description: string;     // 機能の詳細説明（AI生成）
  size: "S" | "M" | "L" | "XL";  // 規模判定 → 人日へ
  isHighRisk: boolean;     // 地雷カテゴリ（決済/外部連携/AI-RAG/規制業種）
}
```
※ 入力メタ（業種・規模・予算・デバイス等）は別途 wizard で保持し、AI出力と合わせて成果物に反映。

### 3.2 金額計算（コード側・決定的）
```
人日   = {S:1.0, M:2.0, L:3.5, XL:6.0}[size]
地雷機能は 人日 ×1.5
開発費 = Σ(人日) × ¥35,000
固定費 = PM ¥150,000（常時）＋ AI判定で {PoC ¥100,000 / UI・UX ¥150,000 / QA ¥100,000}
小計   = 開発費 + 固定費
税込   = 小計 × 1.1
```
- 検算（実装テスト `pricing.test.ts` で固定）: 会員制＋決済＋LINE連携(16人日・固定費フル)=税込¥1,166,000 ／ HP制作(8.5人日・PM+UIのみ)=税込¥657,250

### 3.3 AIプロンプト方針
- 役割: 受託開発の見積もりアシスタント。入力（事業形態・業種・規模・システム説明・詳細・デバイス・予算）から必要機能を **漏れなく洗い出し**、各機能を 分類・説明・規模(S/M/L/XL)・地雷フラグ で構造化。
- 必須ルール:
  - **必要機能を補完**（明示外でも認証・権限・入力検証・管理画面・エラー処理・通知・セキュリティ等を含める）
  - **迷ったら大きめの size**（曖昧＝リスク）
  - **地雷カテゴリは isHighRisk=true**（決済/外部連携/AI-RAG/規制業種）
  - **金額・円は出力しない**（スキーマに無い）
  - tool use で `EstimateRequirement` を強制（型違反は再試行）
  - summary/description は見積書に載る丁寧な日本語品質
- モデル: 既定 **Claude Sonnet 4.6**（品質/コスト/速度のバランス）。品質不足なら Opus に上げる
- 任意: サロン事例などを **1-shot 例** として品質/形式を固定

### 3.4 DB設計（Supabase / Postgres）
**方針**: 個人PJ＆コンソール閲覧優先 → まず **1テーブル `quotes`**（後で正規化可）。
- 入力: `business_type, system_description, industry, employee_size, deployment_target, system_type, detail, ongoing_services, desired_timeline, devices[], budget`
- AI出力: `project_title, summary, features(JSONB), needs_hearing`
- 計算: `dev_man_days, total_excl_tax, total_incl_tax`
- 連絡先(最後に保存): `email, company_name, is_individual, contact_name, phone, agreed_terms`
- 成果物: `quote_pdf_url`（見積書）, `spec_pdf_url`（構想書＝叩き台）（Supabase Storage）
- 営業: `status`（新規/対応中/商談化/失注）← コンソールで更新
- セキュリティ: **RLS有効**（公開APIからは書き込みのみ・読み取り不可）。閲覧はコンソール（サービスロール）

### 3.5 生成パイプライン（AI→金額→見積書PDF→メール）

**Step 1 — 生成（wizard完了時／Server Action）**
1. 全回答を受け取る → Claude（tool use）で `EstimateRequirement` 取得
2. コードで金額計算（人日→¥、地雷×1.5、固定費、税）
3. `quotes` に保存（入力＋AI出力＋計算結果、status=生成済）
4. 「お見積もりが完成しました！」を返す（**金額は画面に出さない**＝メール必須）

**Step 2 — 連絡先取得（メール→お客様情報／Server Action）**
- email・連絡先（会社名/担当/電話/同意）を該当 `quotes` 行に保存、status更新

**Step 3 — 成果物生成＆送付（連絡先確定後）**
同一の quotes データから **2点のPDF** を生成（同じ `EstimateRequirement` を別テンプレで描画）。
1. **見積書PDF**：明細（features×人日×単価）＋ 固定費 ＋ 税 ＋ 注記
2. **プロジェクト構想書PDF（叩き台）**：`summary` を冒頭に、`features` を「ご要望の整理／想定機能（category・name・description）／前提・確認したい点（assumptions・needsHearing）」として描画。**金額・人日・size は載せない**（構想書は要件の面、見積書は金額の面と役割分離）。**末尾に必須注記**「本書は初期構想（叩き台）です。確定要件は詳細ヒアリングのうえ商談で定めます」
3. HTML→PDF（Puppeteer系）で両方レンダリング → **Storage** 保存 → URLを quotes に記録（`quote_pdf_url` / `spec_pdf_url`）
4. **Resend** で2点を添付してメール送付 → サンクス画面。**CC に `sales@tragend.com`（社内控え）** を必ず付ける（顧客送付内容を社内でも確認）。設定値は [company.ts](../../src/lib/estimate/company.ts) `MAIL.salesCc`

> 線引き（RD §3.1）厳守：構想書は **叩き台** であり契約仕様ではない。AI出力にそのまま則って開発しない。確定要件は商談で人間が定める。

**実装メモ**
- PDF生成は重い → 推奨は **送信受付 → バックグラウンド生成 → メール**（タイムアウト回避・UX良）。MVPは同期でも可
- Vercel実行時間制限＋Puppeteerのserverless対応（`@sparticuz/chromium` 等）に注意
- 失敗時フォールバック（「担当よりご連絡します」メール）／重複送信・レート制限
- 見積書テンプレのデザイン・文言は別途作成（Tragend独自）

## 4. セキュリティ方針（顧客資材の保護 ＋ 納品物の安全性）
**(a) 本システム自体（問い合わせ＝顧客資材を扱う）**
- LLM呼び出し・DB操作は **全てサーバ側**（Server Actions）。**APIキーはクライアントに出さない**（.env・Git除外）
- Supabase は **RLS（行レベルセキュリティ）** でリード/案件を保護
- **入力バリデーション＋スパム/レート制限**（フォーム悪用・API課金爆撃の防止）／HTTPS／同意取得済み／データ保持は最小限

**(b) 納品物としてのセキュリティ＝見積もりに織り込む**
- 認証・データ保護・セキュリティ対応は "タダ" にせず **機能として計上**（AIの機能導出に含める）。安全な納品を価格に反映
