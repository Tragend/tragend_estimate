# miryo.ai 参考分析 — サイト動線（flow）

> **目的**: 「ほぼ同じに作る」ための、見積もりサイトの動線・選択肢・結果の記録。
> **運用**: ユーザーがスクショ共有 → Claude が構造化して追記する living document。
> **注意**: フォーマット参考であり丸コピーはしない（文言・ブランド・実績は独自）。出力物分析は `outputs.md` 予定。

- 企業HP: https://miryo.ai/
- 見積もりサイト: https://quote.miryo.ai/

## 全体動線サマリ（v2: 入口〜メール送付まで全判明）

```
quote.miryo.ai（ランディング）
  ├ 入口A: 事業形態[企業・法人 / 個人事業主] → 「作りたいシステム」フリーテキスト
  └ 入口B: ランディング下部フリーテキスト →[今すぐ概算を見る]
        ↓ （businessType ＋ システム説明文 を保持）
  アンケート（全9問・1画面1問ウィザード）
    Q1業種 → Q2従業員規模 → Q3導入先 → Q4システム種類 → Q5詳細(自由文)
    → Q6商談中サービス(自由文) → Q7導入時期 → Q8デバイス(複数) → Q9予算感
        ↓
  「お見積もりが完成しました！」→ AI生成（ローディング）
        ↓
  メールアドレス入力（必須）
  → お客様情報入力（担当者名/電話/会社名or個人/規約同意）
  → 見積書＋要件定義書を【メール送付】＋ 送信完了（サンクス）画面
        ↓
  （日程調整は画面になし。メール内デッキのQRからのみ）
```
※ 入口A/Bは2系統で冗長 →【意図メモ】我々は動線を一本化する方針。

## ★重要な発見1：機能は「選ばせず」AIが自由文から導出している
- アンケートに **「機能を選んでください」という設問は存在しない**。
- 出力デッキの「主要機能」チップ／機能一覧(F-001〜F-015) は、**入口のシステム説明文＋Q5/Q6の自由文から、AIが丸ごと導出**したもの。
- 選択式の設問(Q1〜Q4・Q7〜Q9) は **リード属性＋制約＋提案の文脈**（業種/規模/導入先/種類/時期/デバイス/予算）。
- 出力との対応: プロダクト種別←Q4 ／ 対応端末←Q8 ／ 予算感←Q9 ／ 背景・ゴール←自由文の要約 ／ **主要機能←AI導出**。
- → **Pattern B の核心そのもの**。ユーザーは「作りたいもの」を言葉で言うだけ、機能分解はAIが担う。

## ★重要な発見2：結果は画面に出さず「メール必須」
- 回答完了 → 生成 → **メールアドレス入力が必須**。成果物（見積書/要件定義書）は **画面表示されずメール送付のみ**。
- ＝ 完全なリード獲得設計（メールを出さないと何も得られない）。
- 【設計判断メモ】**メール必須にする（確定：ユーザー指示）**。

## ★重要な発見3：聞く順番が「離脱率最小化」設計
- 詳細な連絡先（担当者名・電話・会社名・規約同意）は **最後の最後**。最初に求めると心理的離脱が増えるため、**価値（見積もり生成）を見せてから**重い情報を取る **プログレッシブ・コミットメント** 設計。
- 順番: 低摩擦の自由文 → タップ中心の9問 →「完成しました！」(報酬) → メール1個 → フル連絡先。
- 【意図メモ】我々もこの順序を踏襲する。

## 技術観察

| リクエスト | 役割（解読） |
|---|---|
| GET `…/system-input?…&_rsc=…`（200・実レスポンスなし） | Next.js の RSCプリフェッチ（遷移高速化）。独自APIではない |
| GET `/api/utils/client-ip` → `{"ip":"…"}` | 訪問者IP取得。リードに付与（分析・スパム検知用と推測） |
| POST `…/system-input`（303）`["corporate","<説明文>",{ipAddress:"$undefined"}]` | **Server Action**。(事業形態, 説明文, IP) でリード下書き作成→アンケートへリダイレクト |
| POST `…/questionnaire/flow/{sessionId}` `[sessionId, questionId, answer]`（例 `["cmqf…","q1","IT・情報通信業"]`） | **Server Action**。各設問の回答をセッションに**逐次保存**（questionId q1..q9 で管理） |
| GET `/api/auth/session`（生成ローディング中） | 認証セッション確認（Auth.js/NextAuth系）。匿名セッション/リード紐付け用と推測 |
| POST `…/questionnaire/contact/{sessionId}` `[sessionId, {email}]`（RSCストリーム応答） | **Server Action**。メール（次いでお客様情報）をリードに保存。応答は参照ID付きのRSC flight形式 |

→ **セッションID(cuid)で1問ずつ回答を蓄積 → 完了で生成 → メール送付** という構造。我々も「セッション＋逐次保存」型でそのまま再現可能（Next.js一致）。

### 最終ステップ（生成後・2画面）: 結果の受け取り＝リード獲得

**(1) メール入力**「**お見積もりが完成しました！**」（進捗バー満タン）
- 質問「見積送付先のメールアドレスを教えてください」／補足「会社用/ビジネス用のメールアドレスをご入力ください。」
- 入力: **メールアドレス（必須）** placeholder `your@example.com`
- 注記「…見積書と要件定義書をお送りします。迷惑メール対策で…受信設定をご確認ください。」

**(2) お客様情報入力**「最後にお客様情報をご入力ください」（メール送信でこの画面へ遷移）
- メールアドレス（前画面の値を表示・「変更する」で編集可）
- **会社名**（placeholder 株式会社○○○）＋「個人」チェック（個人なら会社名不要）
- **担当者名（必須）** placeholder 山田 太郎
- **電話番号（必須）** placeholder 090-1234-5678
- **利用規約・プライバシーポリシー同意（必須チェック）**「送信前に同意が必要」
- → 送信で **見積書＋要件定義書をメール送付**
