## 1. 目的・背景・狙い／ターゲティング／企画

**3つの狙い**
1. 見積もり・問い合わせ対応の自動化によるコスト削減（※問い合わせは未到来。"いずれ"への備え）
2. 「AIを活用したシステム」の自社実績・営業材料づくり（← **今回の主軸**）
3. 仕組みを自作し技術的自信をつける

**ターゲティング**
- 一次対象: IT専任者のいない中小企業・個人事業主で、システム開発を検討中の見込み客
- 想定案件: Webアプリ／業務・会員管理／AI機能付きなど中小規模
- シーン: Tragendサイト来訪者がセルフで「概算見積書」を即取得 → 商談化

**企画・位置づけ**
- 「問い合わせた瞬間に、AIが概算見積書を自動生成して返す」体験自体が、AI活用の実証＝営業材料
- **本システムの核**: ユーザーは作りたいものを自由文で語るだけ。**AIが機能を分解**し、概算見積書を生成する
- miryo の UX/出力構造を踏襲しつつ、Tragend 独自ブランド・トーンで構築（丸コピーしない）

## 2. スコープ・全体フロー

**全体フロー（確定）**

```
入口（事業形態 ＋「作りたいシステム」を自由文）※入口は1本化
 → 質問ウィザード：選択式メタ（業種/規模/導入先/種類/時期/デバイス/予算）＋ 自由文（詳細・補足）
 → AIが自由文から機能を導出・構造化（※機能はユーザーに選ばせない）
 → 概算見積書（PDF）を生成
 → 「完成」提示 → メール必須入力 → お客様情報（担当者名/電話/会社名or個人/規約同意）
 → 成果物をメール送付（＋リード保存）→ サンクス画面
   ※ 日程調整はメール本文のリンクから（任意）
```

**確定した方針（miryo分析より）**
- **入口1本化**（miryoの2系統は冗長なため）
- **機能はAI導出**（選択式の設問は属性・制約・提案の文脈のみ）
- **メール必須のリード獲得型**（成果物は画面に出さずメール送付）
- **プログレッシブ・コミットメント**（価値提示 → 軽い情報 → 重い情報の順で離脱を抑制）

**スコープ**
- 作る: 入口フォーム／質問ウィザード／AI要件抽出・機能導出／概算見積書PDF自動生成／メール自動送付／リード保存
- 今回は作らない: マーケットプレイス・決済／価格の管理画面／類似事例マッチング（将来の Pattern C）／ポートフォリオ連携／日程調整の自前実装（当面メール内リンク）

## 3. 成果物の定義（出力構造）

根っこ＝ AIが導出する **構造化案件要件**（案件メタ ＋ **AI導出の**機能リスト{ 分類, 名称, 詳細 }）。**同一データ**を **2点の成果物** に描画する（見積書＝金額の面、構想書＝要件の面）。

- **成果物①: 見積書（PDF）**
  ヘッダ（宛名・件名・日付・有効期限・総額）＋ 機能明細（数量・金額の列）＋ 集計 ＋ 注記
- **成果物②: プロジェクト構想書（PDF）＝「叩き台」**
  「ご要望の整理／想定機能リスト／前提・確認したい点」を要件定義の体裁で描画。**末尾に必須注記**「本書は初期構想（叩き台）です。確定要件は詳細ヒアリングのうえ商談で定めます」
- **受け取り**: 両方ともメール送付（画面には出さない）

### 3.1 線引き（無料 vs 有料）— 事故防止の核
「何を作りたいか分かっていない見込み客」に対し、**どこまで無料で出すか**を明確化する。

| 段階 | 担当 | 出すもの | 位置づけ |
|---|---|---|---|
| **無料（本システム）** | AI | 見積書＋**構想書（叩き台）** | 「ここまで理解しました」を見せ、**信頼と商談を獲得**するリードマグネット |
| **商談** | 人間 | **確定要件定義** | ここが有料の上流工程。AIの叩き台を土台にヒアリングで詰める |
| **開発** | 人間＋AI | 確定要件に則り構築 | 受注後 |

**鉄則（2点）**:
1. **AI生成の構想書は契約仕様ではない**。必ず「叩き台（ドラフト）」として出し、**確定要件は人間が商談で定める**（見積もりが"概算"なのと同じ構造）。これに **そのまま則って開発しない**（漏れ・取り違え＝炎上の元）。
2. **完成した要件定義書を無料で渡さない**。要件整理＝本来お金を取れる上流工程。無料で完成品を渡すと自社商品の破壊＋他社流用を招く。無料は「**理解の証明としての叩き台**」に留める。

> 2026-06-18 決定の更新：当初「要件定義書は作らない（簡素化）」としたが、AIが既に `EstimateRequirement`（summary＋features）を構造化出力しており、**同一データの別描画で低コストに構想書を追加できる**ため方針変更。提案書（15枚スライド）は引き続き作らない。
> 両PDFテンプレの体裁は **SD** で作成。

## 4. 技術スタックの選定（確定）

| 領域 | 採用 | 備考 |
|---|---|---|
| フレームワーク | Next.js (App Router) | 既存継続。見た目資産（LP/デザイン）は再利用、見積もりフローは作り直し |
| 実行方式 | Server Actions ＋ セッション逐次保存 | miryoと同方式（回答を1問ずつ保存→完了で生成） |
| DB | Supabase (Postgres) | リード・セッション・案件の保存 |
| LLM | Claude | 自由文→機能導出・構造化出力（tool use）。本システムの心臓 |
| メール送付 | Resend | 成果物の自動送付 |
| PDF生成 | HTML/CSS → PDF（Puppeteer系） | 見積書PDF。Tailwind資産を活かせる |
| セッション / 認証 | 匿名セッション（必要に応じ Auth.js） | リード紐付け |
| ホスティング | Vercel | Next.jsと相性最良 |

> **「AI」＝Claude APIの呼び出し**（モデル学習やデータ分析ではない）。Anthropic公式の **TypeScript SDK** で Next.js 内から実行でき、**Python等の別サービスは不要 → 全てTypeScriptで統一**。将来の Pattern C（類似検索）も Supabase `pgvector` で TS のまま可能。

## 5. 非機能の方針・制約・前提

- **方針**: 個人PJ。小さく・速く・安く。完璧さより「動いて見せられる」を優先
- **非機能**: 性能・可用性は最小限。秘密情報（APIキー等）は `.env` 管理＋Git除外（CLAUDE.md セキュリティルール準拠）
- **制約**: 問い合わせ未到来＝実データなし（Pattern C は将来）。法人設立 2026/4 予定
- **前提**: 成果物は"概算"。正式見積は商談で提示（miryo 同様の注記を必ず入れる）
